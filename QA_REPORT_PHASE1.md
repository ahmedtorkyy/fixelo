## 🔴 Critical — Fix Before Launch

### 1. Gemini API Key is the Wrong Format

**File:** `.env.local`

```
VITE_GEMINI_API_KEY=AQ.Ab8RN6I1OV9wWELgtlhLLu4NP1M4Q9cVDMatdHr4hWlKVn91EQ
```

This key starts with `AQ.` — that is **not** a valid Google Gemini API key. Google AI Studio keys always start with `AIza`. This format looks like an AssemblyAI key or a different service entirely.

**Impact:** Every single AI call will fail with an authentication error. The app cannot generate any fix.

**Fix:** Go to [aistudio.google.com/apikey](https://aistudio.google.com/apikey), create a new key, replace the value in `.env.local`. The key should look like `AIzaSy...`.

---

### 2. No `_redirects` File — All Routes Will 404 on Cloudflare Pages

**File:** Missing — needs to be created at `public/_redirects`

The app uses React Router (client-side routing). Cloudflare Pages serves static files. When a user goes directly to `fixelo.com/history` or `fixelo.com/tools`, Cloudflare looks for an actual file at that path, finds nothing, and returns a 404.

**Impact:** Any page except the homepage breaks if the user bookmarks it, shares a link, or refreshes the page.

**Fix:** Create `public/_redirects` with this single line:
```
/* /index.html 200
```

This tells Cloudflare to serve `index.html` for every path and let React Router handle it.

### 3. Favicon References Wrong File

**File:** `index.html` line 5

```html
<link rel="icon" href="/favicon.ico" />
```

The actual file is `public/favicon.svg` — there is no `favicon.ico` anywhere in the project. The browser will silently 404 and show no favicon on the browser tab.

**Fix:** Change to:
```html
<link rel="icon" href="/favicon.svg" type="image/svg+xml" />
```

### 4. Diagnosis Safety Notes Are Generated by AI but Never Shown

**Files:** `src/lib/prompts/diagnosisPrompt.ts` line 25, `src/types/fix.ts`, `src/components/fix/FailureRecovery.tsx`

The AI diagnosis prompt correctly asks for `scriptSafetyNotes` in its JSON response. But `DiagnosisResult` type doesn't include this field, `parseDiagnosisResult` doesn't parse it, and the diagnosis result screen doesn't show it. The AI's safety warnings for the second fix are silently dropped.

**Fix:**

In `src/types/fix.ts`, add the field to `DiagnosisResult`:
```typescript
export interface DiagnosisResult {
  whatSucceeded: string
  whatFailed: string
  whyItFailed: string
  whatNextFixDoesDifferently: string
  nextFixScript: string
  nextUndoScript: string
  scriptSafetyNotes: string  // ADD THIS
}
```

In `src/lib/scriptGenerator.ts`, add to `parseDiagnosisResult`:
```typescript
scriptSafetyNotes: String(json.scriptSafetyNotes ?? ""),
```

In `src/components/fix/FailureRecovery.tsx`, display it after the "What the New Fix Does Differently" card (same pattern as FixResult.tsx already does for the first fix).

## 🟡 Medium — Polish Before Launch

### 5. Filenames Are Messy

**File:** `src/lib/batGenerator.ts`

The `generateFixFilenameFromProblem` function joins the first 3 words with **no separator** and doesn't filter stop words. Results:

| User types | Fix file | Undo file |
|---|---|---|
| My WiFi keeps disconnecting | `FixMyMywifikeeps.bat` | `UndoMywifikeepsFix.bat` |
| My PC is really slow | `FixMyMypcis.bat` | `UndoMypcisFix.bat` |
| Windows Update is stuck | `FixMyWindowsupdateis.bat` | `UndoWindowsupdateisFix.bat` |

"My" appears in the problem AND in the `FixMy` prefix, so it doubles. The undo name `UndoXxxFix.bat` is grammatically weird.

**Suggested fix — replace both filename functions:**
```typescript
export function generateFixFilenameFromProblem(problem: string): string {
  const stopWords = new Set(["my", "is", "a", "an", "the", "it", "i", "not", "really", "keeps", "keeps"])
  const words = problem
    .toLowerCase()
    .replace(/[^a-z0-9\s]/g, "")
    .trim()
    .split(/\s+/)
    .filter(w => !stopWords.has(w))
    .slice(0, 3)
    .map(w => w.charAt(0).toUpperCase() + w.slice(1))
    .join("")
  return `Fix_${words || "MyProblem"}.bat`
}

export function generateUndoFilenameFromFix(fixName: string): string {
  return fixName.replace(/^Fix_/, "Undo_")
}
```

Results: `Fix_Wifi.bat` / `Undo_Wifi.bat`, `Fix_Pc.bat` / `Undo_Pc.bat`. Clean and readable.

## 🔵 Low — Minor Polish

### 8. Desktop Nav Has No "Coming Soon" Hint

In `Header.tsx`, disabled nav items (Tools, Diagnose, Agent, Community) show as plain greyed-out text on desktop with no tooltip or hint. On mobile they show "(Coming soon)". Add `title="Coming soon"` attribute to the disabled `<span>` elements so desktop users get a tooltip on hover.

### 9. Error Banner Doesn't Clear When User Starts Typing

In `Home.tsx`, after an AI error, the red error banner stays visible even when the user starts typing a new problem in the textarea. It only clears when they submit. Low impact but adds noise. Fix: in `ProblemInput`, call a prop `onClearError` whenever the textarea changes.

### 10. `loading` Prop Hardcoded as `false`

In `Home.tsx`, both `<ProblemInput loading={false} />` and `<FailureRecovery loading={false} />` pass false literally instead of the `loading` state from the hook. This works because the `currentStep` state already hides these components during loading, but it's misleading code. Pass `loading` from the hook.
